# YapClean — Документ требований

## 1. Executive Summary & Code Audit

### Общая оценка
Проект представляет собой рабочий MVP с чистой модульной структурой. Код читаемый, логика разделена по ответственности. Для коммерческого релиза требуется устранить ряд критических уязвимостей и архитектурных ограничений.

### Критические проблемы

| # | Компонент | Проблема | Приоритет |
|---|-----------|----------|-----------|
| 1 | `config_manager.py` | API-ключи хранятся в plain-text JSON в `%APPDATA%` — любое приложение или вредоносный код может их прочитать | 🔴 Критично |
| 2 | `hotkey_listener.py` | Реализация через `keyboard` lib работает **только на Windows** и требует прав администратора. Polling loop с `time.sleep(0.05)` — ненадёжный подход | 🔴 Критично |
| 3 | `api_manager.py` | Жёсткая привязка к Groq — нет абстракции провайдера. Добавление OpenAI/Whisper.cpp потребует переписывания | 🟠 Высокий |
| 4 | `clipboard_injector.py` | `keyboard.send("ctrl+v")` ненадёжен в ряде приложений (терминалы, игры, некоторые IDE). Нет fallback | 🟠 Высокий |
| 5 | `main.py` | `check_config_reload()` читает файл каждые 2 секунды в цикле — неэффективно, лучше использовать `watchdog` | 🟡 Средний |
| 6 | `main.py` | Lock-файл через `tempfile` — ненадёжный механизм single-instance на Windows | 🟡 Средний |
| 7 | `settings_ui.py` | `winreg` импортируется напрямую — крашится на macOS/Linux | 🟡 Средний |
| 8 | `api_manager.py` | LLM-рефайн вызывается **только** при включённом переводе. При обычной диктовке текст вставляется сырым без обработки | 🟠 Высокий |
| 9 | `audio_manager.py` | Нет выбора устройства ввода — всегда используется дефолтный микрофон | 🟡 Средний |
| 10 | Весь проект | Нет обработки ошибок сети (timeout, retry) — при падении API приложение молча игнорирует ошибку | 🟡 Средний |

### Что работает хорошо
- Чистое разделение на модули (Audio / API / Hotkey / Injector / Tray)
- Фоновая обработка через `threading` — UI не блокируется
- Система пресетов и кастомных промптов уже заложена
- Интеграция с Notion через trigger-word — оригинальная фича
- Автоматическое определение языка раскладки клавиатуры

## 2. Функциональные требования

### FR-1: Голосовой ввод (Core Pipeline)

**FR-1.1** Приложение должно запускаться в фоне и отображаться в системном трее.

**FR-1.2** Пользователь должен иметь возможность настроить глобальный хоткей для начала/окончания записи (hold-to-talk и toggle-режимы).

**FR-1.3** При нажатии хоткея приложение должно начать запись с выбранного микрофона. Иконка трея должна визуально сигнализировать о записи (красный цвет).

**FR-1.4** При отпускании хоткея (или повторном нажатии в toggle-режиме) запись останавливается и запускается pipeline обработки.

**FR-1.5** Записи должны игнорироваться только при двух условиях одновременно: длительность < 0.2 сек **и** RMS ниже порога тишины. Короткие, но содержательные слова ("Ок", "Запушил", "Апрув", "Сделано") должны обрабатываться корректно. Фильтрация направлена исключительно на системные звуки и случайные щелчки, а не на короткие осмысленные фразы.

**FR-1.6** Обработанный текст должен быть вставлен в активное окно в позицию курсора.

**FR-1.7** Пользователь должен иметь возможность выбрать устройство ввода (микрофон) из списка доступных.

**FR-1.8 (Smart Undo)** После вставки сгенерированного текста пользователь должен иметь возможность нажать стандартный `Ctrl+Z` / `Cmd+Z` и весь вставленный блок текста моментально удалится, возвращая документ в исходное состояние. Реализуется через механизм: перед вставкой сохраняется длина вставляемого текста, после вставки симулируется одно нажатие `Ctrl+Z` которое отменяет всю операцию вставки как единое действие (стандартное поведение Ctrl+V в большинстве приложений).

### FR-2: Транскрибация (STT)

**FR-2.1** Приложение должно поддерживать следующих STT-провайдеров:
- Groq Whisper (cloud, BYOK)
- Deepgram Nova (cloud, BYOK)
- OpenAI Whisper API (cloud, BYOK)
- Whisper.cpp (локально, CPU/GPU)
- faster-whisper (локально, CPU/GPU)

**FR-2.2** Пользователь должен иметь возможность выбрать активного STT-провайдера в настройках.

**FR-2.3** При недоступности облачного провайдера должен быть автоматический fallback на следующий настроенный провайдер.

**FR-2.4** Приложение должно поддерживать языки диктовки: русский, английский, украинский, немецкий, французский, испанский, польский, итальянский.

**FR-2.5 (Mixed Language / Code-Switching)** Для облачных STT-провайдеров, поддерживающих параметр `prompt` (Whisper API, Groq Whisper), при транскрибации должен передаваться контекстный словарь частых IT-терминов и смешанных конструкций (например: `"commit, push, merge, branch, deploy, API, backend, frontend, PR, main, staging"`). Это улучшает первичное распознавание смешанной речи (русский + английские термины) до этапа LLM-обработки.

### FR-3: LLM-прослойка Smart Formatting (КЛЮЧЕВАЯ ФИЧА)

**FR-3.1** После транскрибации текст **всегда** должен проходить через LLM-обработку перед вставкой (не только при переводе, как сейчас).

**FR-3.2** Пользователь должен выбрать активный профиль (Persona) в настройках или через быстрое переключение в трее.

**FR-3.3** Система должна поддерживать следующие встроенные профили:

| Профиль | Описание |
|---------|----------|
| **IT-специалист / Разработчик** | IT-сленг органично, технические термины на правильном английском (API, IDE, framework) |
| **Менеджер / Предприниматель** | Лаконичность, структура, маркированные списки, деловой стиль (action items, дедлайны) |
| **Писатель / Блогер / Маркетолог** | Авторский стиль, эмоциональная окраска, темп речи; читаемость без сухости |
| **Медик / Юрист / Исследователь** | Строго сохранять термины, латынь, аббревиатуры, номера статей; точность важнее стиля |
| **Обычный пользователь (General)** | Натуральный повседневный стиль, чистый грамотный текст |
| **Специалист поддержки (Support)** | Чёткие вежливые формулировки, структурированные шаги решения, профессиональный тон |
| **HR / Рекрутер** | Деловой стиль, чёткость требований и условий, корпоративная лексика |
| **Преподаватель / Тренер** | Педагогический стиль, структурированность, ясность объяснений, методические термины |

**FR-3.4** Системный промпт LLM-прослойки:

```
Ты — универсальный умный редактор текста. Твоя задача — обработать сырой транскрибированный текст, полученный с микрофона, опираясь на выбранный профиль пользователя (Persona), и выдать идеально отформатированный результат для моментальной вставки в активное окно.

Входные данные:
1. [ТЕКСТ] — сырой транскрипт голоса пользователя.
2. [ПРОФИЛЬ] — выбранная пользователем предустановка из настроек приложения.

Общие правила (применяются всегда):
1. Исправь грамматику, синтаксис и расставь знаки препинания.
2. Удали слова-паразиты, заминки, случайные повторы и междометия (э-э, ну, короче, типа).
3. Если мысль меняется или текст длинный, логически разбивай его на абзацы.
4. Выведи ТОЛЬКО отформатированный текст без лишних слов. Запрещены любые вступительные или пояснительные фразы (например, "Вот ваш текст:").

Специфичные правила по профилям (применять строго в зависимости от [ПРОФИЛЬ]):
- IT-специалист / Разработчик: Оставляй IT-сленг органично. Пиши технические термины и аббревиатуры на правильном английском (API, IDE, Mac, framework).
- Менеджер / Предприниматель: Фокус на лаконичности и структуре. Используй маркированные списки для задач. Деловой стиль (action items, дедлайны).
- Писатель / Блогер / Маркетолог: Сохраняй авторский стиль, эмоциональную окраску и темп речи. Улучшай читаемость, не делай текст сухим.
- Медик / Юрист / Исследователь: Строго сохраняй профессиональные термины, латынь, аббревиатуры, номера законов. Точность важнее стиля.
- Обычный пользователь (General): Натуральный повседневный стиль. Чистый грамотный текст без изменения смысла и лексики.
- Специалист поддержки (Support): Чёткие вежливые формулировки. Структурируй шаги решения. Профессиональный тон.
- HR / Рекрутер: Деловой стиль. Чёткость в описании требований и условий. Корпоративная лексика.
- Преподаватель / Тренер: Педагогический стиль. Структурированность. Ясность объяснений. Методические термины.
```

**FR-3.5** Пользователь должен иметь возможность создавать, редактировать и удалять кастомные профили.

**FR-3.6** Пользователь должен иметь возможность редактировать глобальный системный промпт.

**FR-3.7** Приложение должно поддерживать следующих LLM-провайдеров:
- Groq (cloud, BYOK)
- OpenAI GPT-4o-mini / GPT-4o (cloud, BYOK)
- Ollama (локально)
- Облачный провайдер приложения (платная подписка)

**FR-3.8** Режим "Bypass" — возможность отключить LLM-обработку и вставлять сырой транскрипт (для скоростных сценариев).

**FR-3.9 (App-Awareness / Auto-Switching)** Приложение должно определять имя процесса активного окна, в которое будет вставлен текст (через платформенные API: `GetForegroundWindow` на Windows, `NSWorkspace` на macOS, `xdotool`/`wnck` на Linux). Пользователь должен иметь возможность в настройках привязать профиль к имени процесса. Примеры привязок по умолчанию:

| Процесс | Профиль по умолчанию |
|---------|----------------------|
| `Code.exe`, `Cursor.exe`, `idea64.exe` | IT-специалист / Разработчик |
| `Slack.exe`, `Telegram.exe`, `Discord.exe` | Обычный пользователь |
| `WINWORD.EXE`, `Notion.exe` | Менеджер / Предприниматель |
| `chrome.exe`, `firefox.exe` | Обычный пользователь |

При отсутствии привязки применяется активный профиль из настроек. Пользователь может добавлять, редактировать и удалять привязки.

### FR-4: Безопасное хранение ключей

**FR-4.1** API-ключи должны храниться в системном защищённом хранилище:
- Windows: Windows Credential Manager (`keyring` lib)
- macOS: Keychain
- Linux: Secret Service API (через `keyring` lib)

**FR-4.2** В `config.json` должны храниться только неконфиденциальные настройки (хоткей, язык, профиль и т.д.).

**FR-4.3** При первом запуске приложение должно предложить мастер настройки (onboarding wizard) с выбором режима работы (Local / BYOK / Cloud).

### FR-5: Монетизация и лицензирование

**FR-5.1** Приложение должно поддерживать три режима работы:
- **Free / Local** — только локальные модели, без ограничений
- **BYOK** — личные API-ключи, без ограничений по использованию
- **Pro (подписка)** — облачные модели приложения, лимиты по тарифу

**FR-5.2** В режиме Pro приложение должно аутентифицироваться через backend-сервис (JWT/OAuth).

**FR-5.3** Лицензионный ключ или токен должен храниться в системном хранилище (см. FR-4.1).

### FR-6: Интеграции

**FR-6.1** Интеграция с Notion (уже реализована) — сохранить и улучшить:
- Trigger-word для отправки заметки
- Автоматическая категоризация через LLM
- Поддержка нескольких баз данных

**FR-6.2** Выбор устройства ввода (микрофон) из списка доступных в системе.

**FR-6.3** Автозапуск при старте системы (Windows: реестр, macOS: LaunchAgent, Linux: systemd user service).

### FR-7: Кроссплатформенность

**FR-7.1** Приложение должно работать на Windows 10+, macOS 12+, Ubuntu 20.04+.

**FR-7.2** Глобальный хоткей должен работать без прав администратора на всех платформах.

**FR-7.3** Вставка текста должна работать в большинстве приложений на каждой платформе.

### FR-8: Сборка и дистрибуция

**FR-8.1** Windows: `.exe` инсталлятор (NSIS или Inno Setup) + portable `.exe`.

**FR-8.2** macOS: подписанный `.dmg` с нотаризацией Apple. Info.plist должен содержать:
- `NSMicrophoneUsageDescription`
- `NSAccessibilityUsageDescription`

**FR-8.3** Linux: `.AppImage` (универсальный) + `.deb` (Debian/Ubuntu).

**FR-8.4** CI/CD pipeline (GitHub Actions) для автоматической сборки под все платформы при создании тега.

## 3. Нефункциональные требования

### NFR-1: Производительность
- Задержка от отпускания хоткея до вставки текста: **< 3 сек** при облачном STT + LLM
- Задержка при локальном Whisper.cpp (CPU): **< 5 сек** для записи до 30 сек
- Потребление RAM в фоне: **< 150 МБ** (без локальных моделей)
- Потребление CPU в фоне (idle): **< 1%**

### NFR-2: Надёжность
- Приложение не должно крашиться при недоступности сети — graceful degradation
- Retry-логика для API-запросов: 3 попытки с exponential backoff
- Все исключения должны логироваться в файл (`%APPDATA%/YapClean/logs/`)
- Single-instance через именованный mutex (Windows) / lock-файл с PID (Unix)

### NFR-3: Безопасность
- API-ключи никогда не должны попадать в логи
- Конфиг-файл не должен содержать секреты
- Аудио-файлы должны удаляться сразу после обработки (уже реализовано)
- HTTPS для всех API-запросов

**NFR-3.5 (No-Spy Policy)** Приложение принципиально **НЕ делает снимки экрана** и **НЕ читает содержимое активных окон** за пределами получения их системного имени процесса (для App-Awareness). Это требование должно быть явно отражено в архитектуре:
- Запрещено использование библиотек захвата экрана (`mss`, `pyautogui.screenshot`, `PIL.ImageGrab` и аналогов)
- Запрещено чтение буфера обмена без явного действия пользователя
- Запрещена отправка любых данных об активных окнах или содержимом экрана на внешние серверы
- Любой аудит открытого исходного кода должен подтверждать 100% приватность: единственные данные, покидающие устройство — аудиофайл (для облачного STT) и текст транскрипта (для облачного LLM), и только при явном выборе облачного провайдера пользователем

### NFR-4: UX
- Onboarding wizard при первом запуске (< 3 шага)
- Визуальная индикация состояния: idle / recording / processing / error
- Уведомления об ошибках через системные нотификации (не только в консоль)
- Быстрое переключение профиля через контекстное меню трея

### NFR-6: Локализация интерфейса (i18n)
- Язык интерфейса по умолчанию: **английский (en)**
- При первом запуске (onboarding) пользователь выбирает язык интерфейса
- Язык интерфейса хранится в `config.json` и может быть изменён в настройках в любой момент
- Все строки UI вынесены в отдельные файлы локализации (например, `locales/en.json`, `locales/ru.json`)
- Обязательные языки для релиза v1.0:
  - English (en) — по умолчанию
  - Русский (ru)
- Дополнительные языки (v1.x, через community contributions):
  - Українська (uk), Deutsch (de), Français (fr), Español (es)
- Язык интерфейса **не влияет** на язык диктовки и язык LLM-обработки — это независимые настройки

### NFR-5: Кроссплатформенность
- Единая кодовая база для всех платформ
- Платформо-специфичный код изолирован в отдельных модулях
- Нет прямых импортов `winreg`, `ctypes.WinDLL` в общем коде

## 4. Correctness Properties (для Property-Based Testing)

### CP-1: Audio Pipeline
- **CP-1.1** `stop_recording()` всегда возвращает `None` если длительность < `MIN_AUDIO_DURATION_SEC`
- **CP-1.2** `stop_recording()` всегда возвращает `None` если RMS < `SILENCE_THRESHOLD_RMS`
- **CP-1.3** Возвращённый файл всегда существует на диске и является валидным WAV
- **CP-1.4** Файл удаляется после `process_audio()` независимо от результата (finally-блок)

### CP-2: LLM Middleware
- **CP-2.1** Результат `refine_text()` никогда не содержит вступительных фраз ("Вот ваш текст:", "Sure!", "Here is")
- **CP-2.1a (LLM Refusal Fallback)** Если LLM возвращает стандартные отказы или галлюцинации (паттерны: "I'm sorry", "As an AI", "I cannot", "Извините, я не", "Как языковая модель"), `refine_text()` должен вернуть сырой транскрипт без изменений, а не текст отказа. Вставка отказа LLM в рабочий документ пользователя недопустима.
- **CP-2.2** Результат `refine_text()` для непустого входа всегда непустой
- **CP-2.3** При пустом входе `refine_text()` возвращает пустую строку без вызова API
- **CP-2.4** Профиль из настроек всегда применяется к запросу (system prompt содержит название профиля)

### CP-3: Secure Storage
- **CP-3.1** После `save_key(service, key)` вызов `get_key(service)` возвращает то же значение
- **CP-3.2** `config.json` после сохранения не содержит строк, похожих на API-ключи (regex: `^[A-Za-z0-9_\-]{20,}$`)
- **CP-3.3** `delete_key(service)` после выполнения делает `get_key(service)` возвращающим `None`

### CP-4: Provider Adapter
- **CP-4.1** Все провайдеры реализуют единый интерфейс `transcribe(audio_path, language) -> str`
- **CP-4.2** Все LLM-провайдеры реализуют `complete(messages, model) -> str`
- **CP-4.3** Fallback-цепочка провайдеров пробует следующего при исключении предыдущего

### CP-5: Text Injection
- **CP-5.1** Буфер обмена восстанавливается до исходного значения после вставки (даже при исключении)
- **CP-5.2** Пустой текст не вызывает операцию вставки

### CP-6: Notion Trigger
- **CP-6.1** Trigger-word обнаруживается только в начале или конце транскрипта
- **CP-6.2** После удаления trigger-word из текста он не содержит этого слова
- **CP-6.3** При `enable_notion=False` метод `categorize_and_send_to_notion()` не делает HTTP-запросов

## 7. Поэтапный план реализации

### Phase 1: Security & Foundation (Неделя 1-2)
1. Перенести API-ключи в `keyring` (Windows Credential Manager / macOS Keychain / Linux Secret Service)
2. Заменить `keyboard` lib на `pynput` для кроссплатформенного хоткея
3. Изолировать платформо-специфичный код (`winreg`, `ctypes.WinDLL`)
4. Добавить нормальное логирование в файл (модуль `logging`)
5. Исправить single-instance через именованный mutex

### Phase 2: LLM Middleware & Personas (Неделя 3-4)
1. Реализовать Provider/Adapter паттерн для STT и LLM
2. Включить LLM-обработку **всегда** (не только при переводе)
3. Добавить все 8 профилей пользователя
4. Добавить быстрое переключение профиля через меню трея
5. Реализовать режим Bypass (без LLM)
6. Реализовать LLM Refusal Fallback (CP-2.1a) — детектор отказов с возвратом сырого транскрипта
7. Реализовать App-Awareness (FR-3.9) — определение активного процесса и автопереключение профиля
8. Реализовать передачу контекстного словаря в STT prompt (FR-2.5) для улучшения code-switching

### Phase 3: UX & Onboarding (Неделя 5)
1. Onboarding wizard (3 шага)
2. Системные нотификации об ошибках
3. Выбор устройства ввода (микрофон)
4. Визуальный индикатор состояния (idle / recording / processing / error)
5. Smart Undo (FR-1.8) — корректное взаимодействие с Ctrl+Z после вставки
6. UI для управления привязками App-Awareness (процесс → профиль)
7. Система локализации (i18n): вынести все строки в `locales/en.json` и `locales/ru.json`, добавить выбор языка в настройки и onboarding

### Phase 4: Packaging & Distribution (Неделя 6)
1. Windows: PyInstaller → NSIS инсталлятор
2. macOS: PyInstaller → `.app` → `.dmg` с подписью и нотаризацией
