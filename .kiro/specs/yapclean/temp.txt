Разбор фишек Typeless и оценка для YapClean

1. Онбординг и обучение (Интерактивный тур)

Оценка: Это must-have для системного приложения. Пользователи (особенно не гики) часто скачивают утилиту, запускают и не понимают, что делать дальше, потому что у нее нет главного окна, а только иконка в трее.

Сложность: Низкая/Средняя.

Как сделать в YapClean: При первом запуске мы не просто показываем скучный Wizard с галочками. Мы делаем интерактивный туториал:

Шаг 1: "Выбери свой профиль" (Разработчик, Писатель и т.д.).

Шаг 2: "Нажми свой хоткей прямо сейчас и скажи любую фразу". Приложение само распознает речь и вставит ее в тестовое поле прямо в окне онбординга, показывая магию в действии.

Вердикт: Добавляем в PRD (в раздел Phase 3: UX & Onboarding).

2. Внутренний кабинет / Дашборд

Оценка: Для системной утилиты сложный дашборд — это часто overkill. Пользователю нужно просто диктовать текст, а не смотреть на графики. Однако, простая статистика может быть полезна (например, "Сколько слов вы продиктовали" или "Сколько времени сэкономили").

Сложность: Средняя.

Как сделать в YapClean: Сделать минималистичное окно настроек, в котором будет вкладка "Статистика" (Stats). Там можно показывать: сэкономленное время (из расчета, что диктовка быстрее печати в 3-4 раза), количество сказанных слов и текущий выбранный профиль.

Вердикт: Можно добавить базовую статистику (Saved Time) для геймификации, но полноценный дашборд отложим до версии 1.1 или 2.0.

3. Growth- и маркетинг-инструменты (Аффилейт, Пригласи друга, Отзывы)

Оценка: Это золотая жила для инди-проектов без огромных маркетинговых бюджетов. Встроенная рефералка — отличный двигатель продаж.

Сложность: Зависит от реализации биллинга.

Как сделать в YapClean:

Оставить отзыв: В окне настроек или меню трея добавить кнопку "Love YapClean? Rate us" (с ссылкой на Product Hunt, G2 или Gumroad/LemonSqueezy).

Партнерка: Если мы используем LemonSqueezy или Gumroad для приема платежей, у них уже есть встроенный Affiliate-модуль. Мы просто добавляем пункт меню "Become an Affiliate / Earn 30%" с ссылкой на нашу партнерскую программу на этих платформах.

Вердикт: Обязательно добавляем кнопки/ссылки на отзывы и партнерку в меню трея и окно настроек.

4. Контент и сценарии использования (Use cases)

Оценка: Это классика SEO-продвижения и контент-маркетинга. Разные лендинги под разные профессии ("Диктовка для врачей", "Диктовка для программистов") отлично собирают узкоцелевой трафик.

Сложность: Нулевая (это задача для лендинга, а не для кода приложения).

Как сделать в YapClean: На нашем сайте YapClean.tech мы должны создать отдельные страницы/блоки под каждый наш профиль из PRD (FR-3.3):

YapClean for Developers (фокус на IT-сленг)

YapClean for Writers (фокус на авторский стиль)

YapClean for Managers (фокус на Bullet points)

Вердикт: Закидываем эту задачу в бэклог по маркетингу и сайту (к промпту для создания лендинга).

5. Коммуникация о безопасности/надёжности (Safety / Guardrails)

Оценка: Для нашей целевой аудитории (BYOK, локальные модели) безопасность — это главный аргумент против Wispr Flow.

Сложность: Отразить в маркетинге легко, защитить на уровне кода — сложнее.

Как сделать в YapClean:

Маркетинг: На сайте мы уже делаем упор на "No Screen Recording" и "No Clipboard Snooping" (добавили это в лендинг).

Код: Защита от Prompt Injection в контексте нашей утилиты менее критична, так как пользователь сам диктует текст. Однако, мы можем добавить галочку в настройках: "Strict Formatting Mode" (Строгий режим форматирования), который будет заставлять LLM жестко следовать правилам профиля и минимизировать риск галлюцинаций.

Вердикт: В основном это маркетинговая фишка (отражено в манифесте приватности на сайте). В коде мы опираемся на системное безопасное хранилище ключей (FR-4).

FR-1.2 Система мульти-хоткеев (Three-Mode Pipeline) Приложение должно поддерживать три независимых сценария использования, каждый из которых вызывается своим настраиваемым хоткеем. Режим 2: Синхронный перевод (Translate Mode) — Второй хоткей (например, Shift + Alt)

Поведение: Независимо от того, на каком языке говорит пользователь, LLM-прослойка получает жесткую системную инструкцию: "Переведи этот текст на [Выбранный язык, по умолчанию Английский], сохранив профессиональный ИТ-стиль и отформатировав его". Вставляется только готовый перевод.

Режим 3: Голосовой промпт (LLM Command Mode) — Третий хоткей (например, Ctrl + Alt или двойное нажатие)

Поведение: Голос транскрибируется, но транскрипт передается в LLM не для форматирования, а как прямой запрос (Prompt).

Пример: Пользователь говорит: "Напиши функцию на Python для парсинга JSON". Приложение вставляет в IDE готовый код функции, а не транскрипт голоса. Выполни задачу пользователя и выдай ТОЛЬКО результат, без вводных слов (без 'Конечно, вот ваш код'), чтобы его можно было сразу вставить в документ".

https://github.com/taraschernov/v0-yapclean-ui-prototype/pull/1 Готово. Вот что добавлено и почему — кратко по четырём пунктам:

**1. Static export (условный).** В `next.config.mjs` включается через `NEXT_PUBLIC_EXPORT=1 pnpm build` (или `BUILD_TARGET=pywebview`) — тогда Next.js кладёт HTML/JS/CSS в `out/`, которую pywebview открывает с диска. Без флага всё работает как обычно в v0-preview. `trailingSlash: true` нужен, чтобы роуты типа `/pill/` корректно резолвились в `out/pill/index.html` при загрузке через `file://` или статический Flask.

**2. API-слой с fallback на моки.** В `lib/api.ts` — единый клиент: типы (`Config`, `KeysStatus`, `Preset`, `AppRule`, `Mic`), моки (`MOCK_CONFIG`, …), SWR-хуки (`useConfig`, `useKeys`, `useMics`, `usePresets`, `useAppRules`) и сейверы с оптимистичным `mutate`. Если GET падает/таймаутит (800-1500 мс) — возвращается мок; POST в мок-режиме «притворяется успешным». Всё это прозрачно для компонентов.

**3. Base URL — `/api`, с override через env.** Константа `API_BASE = process.env.NEXT_PUBLIC_API_BASE ?? "/api"`. В продакшене pywebview Flask раздаёт и UI, и API с одного `localhost:PORT` — CORS не нужен. Для разработки против удалённого Flask задаёшь `NEXT_PUBLIC_API_BASE=http://localhost:7654/api` в `.env.local`.

**4. Write-only секреты.** `SecretInput` теперь принимает `hasKey?: boolean` — при `true` и пустом input’е показывает плейсхолдер `•••••••• saved` зелёного оттенка. Обёртка `SecretRow` в General-вкладке показывает кнопку **Save** когда поле заполнено, **Clear** когда ключ сохранён и поле пусто. `saveKey(provider, value | null)` шлёт `POST /api/keys` с `{ provider, value }`, где `null` = удалить. Значения ключей с бэкенда никогда не возвращаются.

**Бонусом:** в футере окна настроек теперь живой индикатор статуса: зелёная точка «Connected · changes auto-saved» когда бэкенд пингуется по `/api/health`, янтарная «Backend offline · UI running on mocks» — когда v0-preview или Flask не запущен. Обновили `GeneralTab`, `LlmProfilesTab` и `AppRulesTab` на SWR-хуки + немедленный POST при каждом изменении; `SystemPromptTab` и `StatsTab` без изменений, они работают с локальным состоянием (System Prompt можно подключить к API аналогично, если нужно — дай знать).

**Что должен сделать Python-агент на своей стороне** (для справки):

- Поднять Flask на `localhost:7654`, отдавать статики из `ui/out/` и маршруты `/api/*`.

- Реализовать endpoints: `GET/POST /api/config`, `GET/POST /api/keys` (write-only, хранить через `keyring`), `GET /api/mics`, `GET/POST /api/presets`, `GET/POST /api/app-rules`, плюс `GET /api/health` для пинга.

- pywebview: `webview.create_window(url="http://localhost:7654/")`.
